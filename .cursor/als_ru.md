# Универсальный языковой сервер на базе Free-Foil

## Аннотация
Постановка проблемы.  При реализации языков программирования и систем верификации критически важна корректная подстановка выражений со связанными переменными без захвата имён. Free-Foil решает проблему захвата и автоматически генерирует типобезопасный синтаксис, однако инструменты для быстрой интеграции таких языков в среды разработки отсутствуют.
Цель. Разработать архитектуру и прототип универсального языкового сервера на базе Free-Foil, реализующего ключевые функции Language Server Protocol — подсветка синтаксиса, диагностика, переход к определению, переименование имён и проверка типов при минимальных усилиях со стороны разработчика языка.
Результаты. Предложена модульная архитектура сервера, основанная на обработке запросов через универсальные алгоритмы. Определён набор классов типов (Ranged, FoldablePattern, TypeDeductive и др.), позволяющих активировать функции LSP путём реализации минимального интерфейса для сигнатуры языка и связывателей. Проведён анализ вычислительной сложности: все алгоритмы работают за линейное время от размера дерева в худшем случае. Выявлены ограничения текущей реализации и направления для расширения.
Практическая значимость. Разработанный подход позволяет сократить время создания языкового сервера для нового языка, перенося рутинную работу с обходом дерева и управлением областями видимости в библиотеку. Прототип пригоден для учебных проектов, прототипирования языков и быстрой интеграции доменных языков в современные IDE. Исходный код и документация могут служить основой для дальнейших исследований в области автоматизации инструментов разработки.

## Введение

Проблема захвата имён при подстановке выражений со связанными переменными — фундаментальное препятствие в реализации языков программирования и систем формальной верификации. В общем случае индексы де Брёйна \cite{deBruijn1972} полностью исключают захват за счёт замены имён на числовые индексы, но ценой потери читаемости и необходимости обхода дерева для пересчёта индексов при подстановке. 
В языках со статической типизацией, особенно в Haskell с его развитой системой типов, появляются дополнительные возможности: гарантии корректности работы со связанными переменными можно кодировать на уровне типов, перенося часть проверок на этап компиляции. В этом контексте библиотека bound \cite{bound} и подход внутренне ограниченных типов \cite{BirdPaterson1999} кодируют область видимости в типах данных, но требуют явной манипуляции переменными и ручного описания рекурсивных структур для каждого языка. 
С другой стороны, в некоторых промышленных компиляторах применяется алгоритм Rapier \cite{haskellCompilerSecrets} — stateless-реализация конвенции Барендрегта \cite{Barendregt}. Однако в Rapier корректность работы с областями видимости не проверяется на уровне типов, что, как отмечают разработчики языка Dex, на практике приводит к трудно уловимым ошибкам захвата имён. Библиотека Foil \cite{foil2023} устраняет этот недостаток, представляя собой типобезопасный аналог Rapier: она объединяет однопроходный алгоритм подстановки с гарантиями системы типов через индексацию областей видимости фантомным параметром S. Тем не менее, Foil по-прежнему требует ручного описания синтаксических структур. Наконец, Free-Foil \cite{freefoil2024} решает и эту проблему, автоматически генерируя типобезопасный, эффективный синтаксис с подстановкой, избегающей захвата, на основе декларативной сигнатуры языка.
Free-Foil параметризует абстрактный синтаксис двумя сущностями: типом связывателя binder и бифункторной сигнатурой sig, которая различает обычные узлы и узлы с расширенной областью видимости. Абстрактное синтаксическое дерево строится из двух взаимно рекурсивных типов:
data ScopedAST binder sig n where
    ScopedAST :: binder n l -> AST binder sig l -> ScopedAST binder sig n
data AST binder sig n where
    Var :: Foil.Name n -> AST binder sig n
    Node :: sig (ScopedAST binder sig n) (AST binder sig n) -> AST binder sig n
ScopedAST представляет узел, связанный с новым именем (например, тело лямбды); AST — обычный узел дерева, где сигнатура sig определяет, какие дочерние элементы могут иметь расширенную область видимости.
Все сущности индексируются типом S, представляющим текущую область видимости:
Name (n :: S) — имя, принадлежащее области n. Представлено машинным целым числом.
Scope (n :: S) — область видимости, содержащая имена типа Name n. Представлен набором целых чисел. Поддерживает быстрое сравнение и проверку принадлежности:
если scope1, scope2 :: Scope n, то scope1 == scope2;
если x :: Name n и scope :: Scope n, то x `member` scope == True.
NameBinder (n :: S) (l :: S) — связыватель (паттерн), расширяющий область n до l добавлением одного нового имени.
В качестве демонстрационного примера рассмотрим просто типизированное лямбда-исчисление (λ-исчисление). Его синтаксис опирается на три базовые конструкции: переменные, аппликацию и λ-абстракцию. Особенность языка заключается в работе со связанными переменными: тело λ-абстракции расширяет текущую область видимости новым именем, а при подстановке выражений необходимо гарантировать отсутствие захвата свободных имён. В терминах Free-Foil синтаксис λ-исчисления задаётся бифункторной сигнатурой:
data ExprF scope term
    = AppF term term      -- оба поддерева остаются в текущей области
    | LamF scope          -- тело абстракции переходит в расширенную область
type Expr = AST NameBinder ExprF
Параметр term соответствует дочерним узлам, сохраняющим текущую область видимости (AST), а scope — узлам, чья область расширяется связывателем (ScopedAST). В результате тип Expr n автоматически получает корректную индексацию: свободные переменные имеют тип Name n, а тело λ-абстракции — тип Expr l, где l является расширением n, созданным связывателем NameBinder n l. Система типов Haskell статически запрещает некорректные операции с переменными из несовместимых областей. Становятся доступными подстановка, приведение к нормальной форме (NF) или приведение к слабой головной нормальной форме (WHNF), исключающие необходимость ручного переименования или пересчёта индексов при обходе дерева.
Для бифункторной сигнатуры возможно реализовать Bifoldable, чтобы применить функцию свёртки к узлу: bifoldl (\(ScopedAST binder ast) -> …) (\ast -> ...) initialValue sig. Таким образом можно рекурсивно обойти всё абстрактное синтаксическое дерево (АСД) для произвольных binder и Bifoldable sig, что сделает алгоритм универсальным. Если же для реализации некоторого метода необходимо вывести данные из сигнатуры, для неё можно потребовать соответствующий класс типа. 
Универсальные алгоритмы при работе с абстрактным синтаксисом, представленным с помощью Free-Foil, можно реализовать для языкового сервера, получив в результате универсальный языковой сервер для любых binder и sig, но с требованиями реализовать для них классы типов с функционалом, недоступным в синтаксисе первого порядка.
Автоматизация языковых серверов является актуальной проблемой с ростом предметно-ориентированных языков и потребности в быстрой их интеграции со средами разработки. Это также могло бы помочь учебным проектам и при проектировании новых языков программирования.
Language Server Protocol (LSP) \cite{lsp} — это стандартизированный протокол, определяющий коммуникацию между языковым сервером и интегрированными средами разработки. Несмотря на то, что LSP был представлен корпорацией Microsoft, это продукт с открытым исходным кодом, который поддерживает любая популярная среда разработки. Для Haskell самой популярной библиотекой, реализующий коммуникацию с помощью LSP, является haskell/lsp \cite{lsp-haskell}. С помощью него написан в том числе языковой сервер для Haskell. В представленной работе функционал языкового сервера основывается на этой библиотеке.
Целью работы является разработка универсального языкового сервера на базе Free-Foil, реализующего подсветку синтаксиса, демонстрацию ошибок, предупреждений и сообщений, простую проверку типов, переход к месту определения и переименование имён для среды разработки, использующей Language Server Protocol.

## Реализация
Поведение языкового сервера определяется списком обработчиков. Таким образом, в состав универсального языкового сервера входят обработчики, работа с кэшем и некоторая конфигурация для запуска библиотеки. Для конфигурации используется объект LSConfiguration:
data LSConfiguration binder sig = LSConfiguration
    { fileExtension :: String
    , buildAsts :: String -> [SomeScopeWithAST binder sig] }
data SomeScopeWithAST binder sig where
    SomeScopeWithAST :: Distinct n 
        => Scope n -> AST binder sig n -> SomeScopeWithAST binder sig
При инициализации языковой сервер собирает содержимое файлов с расширением fileExtension и переводит их в абстрактные синтаксические деревья. Поскольку не для любого языка программирования программу удобно представлять как выражение с одной областью видимости для каждого ребёнка, для файла сохраняются несколько деревьев. Перевод из текста в дерево реализует пользователь, поскольку лексер, парсер и конвертация в АСД Free-Foil — это специфичный для языка функционал. Полученные деревья сохраняются в кэше, реализованном с помощью Software Transactional Memory (STM) \footnote{\url{https://hackage.haskell.org/package/stm}}. В среде языкового сервера сохранён словарь для быстрого поиска деревьев в зависимости от пути файла. Вместе с каждым деревом пользователь сохраняет начальную область видимости (Scope), соответствующую данному дереву. Это далее понадобится в алгоритмах, следящих за изменениями области видимости. Далее деревья обновляются при изменении открытых в среде разработки файлов.
При наведении на текст в среде разработки срабатывает соответствующий обработчик, и для него библиотека предоставляет информацию об открытом файле и позиции курсора. Методом, описанным во введении, можно обойти узлы дерева, включающие данную позицию, и найти самый узкий. Узлы, не включающие искомую позицию, исключаются из проверки, но это также означает, что диапазон внешнего узла должен включать диапазоны его детей. Если некоторый узел не может предоставить информацию о своём диапазоне, поиск не продолжит проверять его детей. Поскольку информацию о диапазоне возможно сохранить только в узлах абстрактного синтаксиса второго порядка, например, в аннотациях к узлам, самым узким может стать только Node, в сигнатуре которого уже могут храниться переменные абстрактного синтаксиса первого порядка (Var). 
Реализация перехода к месту определения использует описанный алгоритм. Срабатывает обработчик TextDocumentDefinition, для которого аналогично известно в каком файле и для какой позиции произведён запрос. Если самый узкий Node с подходящей позицией ссылается на узел сигнатуры, содержащий одну Var, которая хранит искомый Name, можно найти связыватель, который его внедряет. Однако нет уверенности, что диапазон найденного узла совпадает с диапазоном вложенной переменной. Также найденный узел может либо не содержать переменных, либо иметь несколько. Решить как конкретно действовать в данной ситуации можно только исходя из сигнатуры, что делает алгоритм менее универсальным. Проблема легко решается дополнительным узлом в абстрактном синтаксисе второго порядка, используемым исключительно для Var. Если пользователь гарантирует, что в АСД Free-Foil каждая переменная находится в некотором узле, где она — единственный ребёнок, можно гарантировать, что, сверив узел по позиции, можно достать одну переменную и подобрать связыватель для Name, который она представляет. Таким образом, реализация перехода к месту определения требует реализации класса типа Ranged для некоторой сигнатуры sig:
class Ranged sig where
    range :: sig (ScopedAST binder sig n) (AST binder sig n) -> Maybe Range
Поскольку для каждого сохранённого в кэше дерева известен начальный Scope, можно для каждого ScopedAST расширять область видимости представленным связывателем и делать проверку на member для искомого имени, пока не будет найден узел, внедряющий искомое имя. 
Однако, если использовать стандартный withFresh для создания связывателей во Free-Foil АСД, возможны коллизии имён в параллельных областях видимости. Например, для пары с двумя лямбдами — (λx.x, λy.y) — каждая из лямбд будет расширять область видимости одним и тем же именем. Так происходит, потому что Scope в стандартной библиотеке представлен набором целых чисел, и для его расширения берётся максимальное значение в наборе и увеличивается на один, каким образом это было реализовано ещё в Foil \cite{foil2023}. 
Решить эту проблему можно, если вместо стандартного Scope использовать иную реализацию, например, OffsetScope, хранящую не только набор переменных в текущей области видимости, но ещё и сдвиг для имён, занятыми параллельной веткой. Но данное решение требует переписать extendScopePattern и withFresh, необходимые для построения АСД, а также делает параллельные узлы сигнатуры неявно связанными. При этом, в дальнейшем для дерева можно продолжать использовать стандартный Scope, и, например, функции, используемые для безопасной подстановки, смогут работать со связывателями, созданными на основе OffsetScope. 
В контексте же универсального языкового сервера каждое новое требование к АСД делает программу специфичнее, поэтому лучше взглянуть на другое решение, лежащее на поверхности. Если для данной позиции уже был найден Name, значит «по пути» к нему в дереве был некоторый ScopedAST, внедряющий данное имя и находящийся на искомой позиции. Иначе описанный ранее алгоритм поиска наиболее узкого узла не смог бы рассмотреть детей предполагаемого узла. Значит узел можно просто сверить по позиции и быть уверенным, что для любого параллельного узла данная проверка провалилась бы, при условии, что пользователь корректно реализовал метод range для сигнатуры.
После нахождения связывателя, внедряющего искомое имя, необходимо определить диапазон, куда переместится и который подсветит среда разработки. Для связывателя должен быть определён класс:
class FoldablePattern pat where
    foldrPat :: (SomePattern pat sig -> r -> r) -> r -> pat n l -> r
    rangePat :: pat n l -> Maybe Range
    binderOf :: pat n l -> Maybe (NameBinder n l)
Поскольку связыватели имеют тип S -> S -> *, реализовать экземпляр Data.Foldable для них нельзя. Предполагается, что связыватели представлены алгебраическим типом данных и имеют вложенную структуру, где на каждом уровне может быть либо ноль, либо один NameBinder, расширяющий область видимости одним именем. Связыватели могут храниться в списке или наборе, но каждый элемент коллекции должен быть узлом алгебраического типа данных, которым представлен связыватель. С одной стороны это делает языковой сервер специфичнее, но с другой предоставляет инструментарий для реализации новых алгоритмов без необходимости создания новой функции для каждого конкретного сценария. Можно применить подход, использованный для узлов АСД, в том числе и к связывателям. Например, рекурсивно обойти структуру и найти узел с подходящей позицией, либо найти первый узел, внедряющий данное имя, и достать его позицию. Второй случай как раз подходит, чтобы предоставить языковому серверу ответ на запрос TextDocumentDefinition.
Переименование имён во многом повторяет переход к определению. Для неё используется обработчик TextDocumentRename. Теперь крайне полезной оказывается возможность свернуть связыватель и проверить каждый отдельный узел отдельно по диапазону, чтобы найти имя, которое требует переименования. Есть два сценария:
Переименовывают имя в узле. Тогда надо сначала убедиться, что действительно есть некоторый Var, который лежит на данной позиции. У переменной можно достать Name, а для него найти связыватель подходом, описанным выше для функции перехода к определению.
Переименовывают имя в связывателе. Значит манипуляции с деревом из первого пункта можно не делать — и связыватель, и узел, область видимости которого он расширил, находятся в одном ScopedAST, известном при первом же обходе дерева.
Далее для узла с расширенной областью видимости необходимо выполнить поиск всех переменных, ссылающихся на данное имя, и для каждого узла достать его диапазон. Необходимо смотреть на паттерн вложенностей Node -> SomeSigForVar -> Var -> Name, где SomeSigForVar — узел сигнатуры, оборачивающий переменные. Совершенно неважно какой это узел, необходимо только, чтобы его диапазон совпадал с диапазоном лежащего внутри Name, по которому сверяется эквивалентность имён при переименовании. Для переименования надо лишь предоставить список диапазонов, которые будут выделены средой разработки для редактирования.
При проверке типов предполагается, что пользователь сохранил бы информацию о типе узлов в сигнатуре, например, в аннотации к узлам. По сути это единственное, что может поменяться, потому что прямого доступа к кэшу у пользователя нет. Поэтому для двунаправленной проверки типов дерево обходится с помощью функции свёртки, где для каждого узла сигнатуры sig необходимо сформировать сигнатуру sig’ с информацией о типах. При этом, поскольку sig — бифунктор, в оригинальной сигнатуре узлы заменены функциями для запуска проверки типа для предполагаемого узла. Для ScopedAST заодно передан тип, указанный в связывателе, что может помочь при выводе типов. Функция проверки типа также для удобства возвращает пару из узла абстрактного синтаксиса первого порядка, который станет ребёнком в узле новой сигнатуры, а также тип, выведенный для него. Таким образом, пользователю необходимо указать эвристику при проверке типов, передав ожидаемые типы для каждого ребёнка, а потом собрать узел с выведенной информацией о типе.
class TypeDeductive sig sig' ty binder where
    deduceType 
        :: ty 
        -> sig 
            (ty, ty -> (ScopedAST binder sig' n, ty)) 
            (ty -> (AST binder sig' n, ty)) 
        -> sig' (ScopedAST binder sig' n) (AST binder sig' n)
sig’ должен соответствовать классу типа Typed:
class Typed sig ty where
    ty :: sig (ScopedAST binder sig n) (AST binder sig n) -> ty
А связыватели должны уметь предоставлять информацию о типах, а также расширять словарь из типов, указанных для имён. Таким образом, автоматизируется сценарий вывода типа для переменной.
class TypedPattern pat ty where
    patTy :: pat n l -> ty
    addPattern :: pat n l -> NameMap n ty -> NameMap l ty
Сообщения, демонстрируемые при наведении курсора на узел, реализуются с помощью обработчика TextDocumentHover. Для данной позиции по аналогии с методами ранее находится самый узкий узел, для которого ожидается, что определён класс типа Hoverable с методом hoverData :: sig (F.ScopedAST binder sig n) (F.AST binder sig n) -> [String]. Для связывателей при этом ожидается определённым класс типа HoverablePattern с методом hoverDataPat :: pat n l -> [String]. Строки, полученные для узла, соединяются в маркированный список и передаются языковому серверу.
Сообщения с диагностикой формируются отдельно от сообщений по наведении. Они бывают разных типов и содержат дополнительную информации об ошибке, поэтому пользователь формирует объекты с сообщением самостоятельно в diagnose :: sig (F.ScopedAST binder sig n) (F.AST binder sig n) -> [Diagnostic], определённом для класса типа DiagnosableSig. Сообщения, полученные для каждого узла каждого дерева в файле, передаются языковому серверу.
Подсветка кода реализована с помощью классов типа TokenizableSig и TokenizablePattern:
type ASTTokens = [SemanticTokenAbsolute]
type ScopedASTTokens = (ASTTokens, ASTTokens)
class TokenizableSig sig where
    tokenize :: sig ScopedASTTokens ASTTokens -> [SemanticTokenAbsolute]
class TokenizablePattern pat where
    tokenizePat :: pat n l -> [SemanticTokenAbsolute]
Для каждого узла АСД необходимо определить в каким порядке будут объединены токены с абсолютным позиционированием, полученные от детей узла, а также добавить произвольные токены, например, для ключевых слов, не представленных в АСД, но сохранённых в сигнатуре. За порядком следить принципиально важно, поскольку абсолютные токены переводятся в относительные для языкового сервера, и диапазоны, начинающиеся раньше самого дальнего предыдущего, пропускаются. Конечно, можно объединять списки с помощью функции свёртки, поскольку результат — семигруппа, но тогда список пришлось бы сортировать. Выбор был сделан в пользу решения с bimap как более универсального — при желании пользователь может отсортировать список самостоятельно.

## Обсуждение
Данная работа стремится создать универсальный языковой сервер на базе Free-Foil, фокусируясь на максимальной автоматизации заданного набора функций. Чтобы минимальным количеством кода, имея АСД Free-Foil, можно было получить языковой сервер со всеми поддерживаемыми и корректно работающими функциями. Вопросы масштабируемости и производительности были менее критичны. Подобный проект демонстрирует возможности и ограничения при реализации универсальных алгоритмов для Free-Foil, а также может стать основой для более продвинутых разработок. 
Для оценки уровня автоматизации сложно посчитать сколько строк кода было сэкономлено при реализации языкового сервера полностью вручную и с использованием Free-Foil, поскольку способов написать один и тот же функционал бесконечно много. Объективно же можно посчитать сколько функций требуется реализовать пользователю, чтобы активировать каждую из возможностей языкового сервера. Более того, пользовательские функции можно поделить на три категории:
те, что работают с сигнатурой языка;
те, что работают со связывателями;
остальные.
Чем чаще пользователю приходится иметь дело с функционалом, касающегося сигнатуры и связывателя, и меньше с Free-Foil, тем больше работы автоматизировала представленная библиотека, поскольку автоматизация достигается благодаря переносу операций с Free-Foil в библиотеку.
Также можно посчитать вычислительную сложность каждой из функций языкового сервера для K узлов всего с максимальной вложенностью L для худшего случая — это может дать представление о производительности.
Парсинг и построение АСД Free-Foil без проверки типов — функция, необходимая для запуска языкового сервера, требующая O(K) операций.
Переход к определению — функция, требующая Ranged sig и FoldablePattern binder, то есть определения 4 функций. Для поиска самого узкого узла необходимо сделать O(L) проверок и потом для поиска связывателя ещё O(L) проверок. Получается O(2×L) = O(L).
Переименование имён — функция, аналогично требующая Ranged sig и FoldablePattern binder. Поиск самого узкого узла — O(L) проверок. Поиск связывателя — ещё O(L) проверок. Затем поиск переменных, ссылающихся на найденное имя, требует O(K). Таким образом, O(2×L+K) = O(K), поскольку K > L.
Проверка типов — функция, требующая TypeDeductive sig sig’ ty binder, Typed sig ty и TypedPattern pat ty. Суммарно 4 функции. Сложность алгоритма — O(K).
Сообщения при наведении — функция, требующая Hoverable sig, HoverablePattern binder, Ranged sig и FoldablePattern binder. Необходимо реализовать 6 функций. Нахождение самого узкого узла — O(L) операций, таким образом, сложность алгоритма — O(L).
Диагностические сообщения — функция, требующая DiagnosableSig sig, за которой скрывается одна функция. Вычислительная сложность — O(K) .
Подсветка кода — функция, требующая TokenizableSig sig и TokenizablePattern binder. 2 функции. Вычислительная сложность — O(K).
Можно сказать, что ни один из алгоритмов в худшем случае не выполняется аномально долго и время на выполнение самых ресурсоёмких задач растёт линейно в зависимости от количества узлов в АСД. При этом наличие всего одной функции (построение АСД Free-Foil), не связанной ни с сигнатурой, ни со связывателем, является индикатором того, что значительную часть логики языкового сервера и работы с Free-Foil при манипуляциях с деревом удалось автоматизировать. Реализация классов типа Ranged sig и FoldablePattern binder открывает сразу две непростые возможности языкового сервера, что делает их рекордсменами по автоматизации. Для использования же всех возможностей инструмента необходимо реализовать 13 функций, работающих с сигнатурой языка или связывателем.
Для текущей реализации универсального языкового сервера возможны доработки в плане функциональности и производительности, а также актуальны следующие замечания:
Сложность поиска имени по позиции может быть константной, благодаря использованию словаря интервалов (IntervalMap).
При поиске самого узкого узла по позиции и связывателя, внедряющего его, возможно делать один проход дерева — идя вглубь искать имя по позиции, а на обратном пути проверять связыватели для найденного имени. 
Библиотека не рассматривает области видимости типов, например, для типов, объявленных пользователем.
Проверка типов выводит тип для переменной только исходя из типа в связывателе, что не позволяет реализовать более сложный алгоритм вывода типов, где аннотации типов переменных опциональны, что актуально для большого количества современных языков программирования. 
Проверка типов могла бы быть более продвинутой и автоматизированной, если бы использовала алгоритмы вывода типов, основанные на унификации ограничений типа.
Библиотека обновляет кэш для всех открытых файлов из запущенной сессии при внесении изменений в код. Построение дерева зависимостей позволило бы оптимизировать обновления, а также идентифицировать циклические зависимости.
Библиотека может производить проверку типов только для изменённых узлов дерева, оптимизируя таким образом обновления при внесении изменений в код.
В библиотеке не предусмотрена работа с глобальным пространством имён. Таким образом, переход к определению не сможет переместить пользователя в другой файл, если имя объявлено там, а прекратится с ошибкой.
Некоторые методы, например, подсветка, могут быть реализованы для общего случая и предложены пользователю как универсальное решение для ещё большей доступности функционала языкового сервера. В случае подсветки кода, имея Ranged sig и FoldablePattern binder, реально подсветить переменные в узлах и связывателях без вмешательства пользователя.

## Заключение

Представленная работа демонстрирует возможности и трудности при реализации универсальных алгоритмов на базе Free-Foil на примере универсального языкового сервера. Разделение абстрактного синтаксиса на первичный и вторичный, а также репрезентация имён и связывателей в абстрактном синтаксисе первого порядка позволяют достичь высокого уровня автоматизации — определив лишь диапазоны для каждого узла сигнатуры и связывателя становится возможной переход к определению переменных и их переименование. 
Работу можно использовать для молодых предметно-ориентированных языков, учебных проектов или при проектировании новых языков программирования, сокращая время разработки языкового сервера. Представленные в работе замечания освещают возможные доработки и улучшения для будущих попыток автоматизировать реализацию языкового сервера.
